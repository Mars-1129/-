## 2. 缺陷管理目标与原则

### 2.1 管理目标
1. 所有缺陷都能追踪到需求、模块、接口、数据对象和责任边界。
2. 所有高优先级缺陷都具备清晰复现路径与回归闭环。
3. 所有长任务问题都能定位到 `task_id`、`trace_id` 和 `current_stage`。
4. 所有回归测试都能说明影响范围、执行结果和遗留风险。

### 2.2 管理原则
1. 现象、原因、影响面、修复状态必须分开记录。
2. Bug 台账不是简要备注，而是正式交付记录。
3. P0 缺陷必须判断是否阻断主链路、演示、答辩和版本交付。
4. 所有修复完成的缺陷必须进入回归，不得以“代码已改”代替回归结论。
5. 关联模块和关联链路必须同步验证，不能只复测单点。

## 3. 缺陷分级标准

### 3.1 严重级别定义
| 级别 | 定义 | 典型场景 | 是否阻断发布 |
|---|---|---|---:|
| Blocker | 主链路不可用，系统核心能力中断 | 无法上传、无法生成剧本、无法成片、无法导出 | 是 |
| Critical | 关键能力异常，结果错误或高风险不一致 | 成片错误、状态机错乱、任务丢失、严重数据不一致 | 是 |
| Major | 重要功能异常，但存在绕过方式 | 分镜编辑异常、局部重渲染错误、看板关键指标错误 | 视情况 |
| Minor | 非核心能力问题，不影响主链路完成 | UI 提示错误、弱提示缺失、边角兼容性问题 | 否 |
| Suggestion | 优化项或体验改进建议 | 文案不一致、交互可优化、字段展示可优化 | 否 |

### 3.2 优先级定义
| 优先级 | 含义 | 处理要求 |
|---|---|---|
| P0 | 必须优先处理 | 立即确认、立即修复、立即回归 |
| P1 | 需在当前版本处理 | 排期修复并完成回归 |
| P2 | 可在后续版本处理 | 记录风险并纳入迭代 |

### 3.3 阻断发布判定规则
以下任一情况默认阻断发布：
1. 无法完成素材→剧本→创作→导出的主链路。
2. `task_id` 不可追踪，或刷新后任务不可恢复观察。
3. 导出成片与任务状态不一致。
4. `product_id` 上下文穿透，返回了其他商品数据。
5. 数据写入 PostgreSQL、Qdrant、MinIO 发生严重不一致。
6. P0 接口错误码、状态机、字段约束与文档定义不一致。

## 4. 缺陷字段规范

每条缺陷记录至少包含以下字段：

| 字段 | 必填 | 说明 |
|---|---:|---|
| Bug 编号 | 是 | 统一编号，例如 `BUG-20260523-001` |
| 标题 | 是 | 一句话说明现象 |
| 所属模块 | 是 | 素材/剧本/创作/分析/公共 |
| 需求来源 | 是 | 对应 PRD/测试用例/接口/数据对象 |
| 严重级别 | 是 | Blocker/Critical/Major/Minor/Suggestion |
| 优先级 | 是 | P0/P1/P2 |
| 环境 | 是 | 本地/测试/演示环境 |
| 前置条件 | 是 | 复现前必要状态 |
| 复现步骤 | 是 | 可重复执行的步骤 |
| 预期结果 | 是 | 正确行为 |
| 实际结果 | 是 | 实际异常现象 |
| 影响范围 | 是 | 用户、模块、链路、环境影响 |
| 接口路径 | 否 | 相关 API 路径 |
| 请求参数 | 否 | 关键入参 |
| 响应体/错误码 | 否 | 关键返回 |
| task_id | 长任务必填 | 长任务追踪键 |
| trace_id | 长任务/链路问题必填 | 跨链路追踪键 |

## 5. 缺陷状态流转规范

### 5.1 标准状态流
```text
New → Confirmed → In Progress → Fixed → Ready for Regression → Closed
                                  └──────────────────────────────→ Reopen
```

### 5.2 状态说明
- New：首次发现，尚未确认
- Confirmed：已复现并确认问题存在
- In Progress：正在修复中
- Fixed：开发已完成修复
- Ready for Regression：可进入测试回归
- Closed：回归通过，正式关闭
- Reopen：回归失败或线上复现，再次打开

### 5.3 状态流转要求
1. 未复现的问题不得直接进入修复。
2. 未回归的问题不得直接关闭。
3. 回归失败的问题必须重新打开并补充最新现象。
4. 长任务类问题在 Reopen 时必须补充新的 `task_id`/`trace_id`。

## 6. 回归测试策略

### 6.1 回归层级
1. 缺陷点回归：只验证当前缺陷是否修复。
2. 同模块关联回归：验证当前模块其他相关能力是否受影响。
3. 主链路冒烟回归：验证素材→剧本→创作→导出主链路是否受影响。

### 6.2 回归触发规则
以下情况必须执行主链路回归：
- 修复 P0 缺陷
- 修复任务状态机缺陷
- 修复导出、SSE、任务恢复、对象存储、数据一致性类缺陷
- 修复 `product_id` 上下文隔离问题
- 修复创作任务阶段推进问题

以下情况至少执行同模块关联回归：
- 接口字段变更
- 错误码变更
- 分镜逻辑变更
- 向量检索逻辑变更
- 模板与爆款仿写逻辑变更

### 6.3 回归结果判定
- Pass：缺陷点与关联链路均通过
- Fail：缺陷仍存在或引入新问题
- Partial：缺陷主现象修复，但存在遗留风险或部分链路异常

## 8. 高频缺陷类型清单

### 8.1 上传与切片类
- 文件大小校验失效
- 格式校验失效
- 素材成功入库但对象存储缺失
- 切片时长越界
- 切片数量与数据库记录不一致

### 8.2 剧本结构与配时类
- 剧本字段缺失
- 分镜时长越界
- 总时长超过 15 秒
- JSON Patch 更新错位
- 配时校验失效
- 合规拦截漏拦或误拦

### 8.3 创作与导出类
- 任务不进入队列
- 阶段顺序错乱
- Cache Miss 不触发兜底
- 局部重渲染误触发全量重渲染
- 导出成功但 `video_url` 无法访问
- 成片音视频不同步

### 8.4 SSE 与任务恢复类
- SSE 不推送
- 进度不更新
- 页面刷新后丢失任务
- `task_id` 查询不到任务
- `trace_id` 不贯通

### 8.5 数据与看板类
- Qdrant point 缺失
- PostgreSQL 与对象存储不一致
- 看板返回空数据无兜底
- A/B 数据混乱

## 11. 缺陷记录示例规范

### 11.1 长任务类缺陷示例
- 标题：创建任务后 SSE 不推送阶段事件
- 模块：创作/任务管理
- 需求来源：长任务状态追踪、SSE 订阅
- 关键字段：
  - `task_id`
  - `trace_id`
  - `current_stage`
  - `status`
  - `progress`
- 影响面：前端无法观察任务执行进度，可能误判任务失败
- 回归要求：
  - 复测 SSE 推送
  - 复测任务详情查询
  - 复测刷新恢复
  - 复测任务完成事件

### 11.2 接口类缺陷示例
- 标题：上传接口缺少 `product_id` 时返回错误码不正确
- 模块：素材
- 接口：`POST /api/v1/materials/upload`
- 应记录：
  - 请求体
  - HTTP 状态码
  - 错误码
  - 返回字段
- 回归要求：
  - 复测同接口其他参数校验
  - 复测成功上传场景
  - 复测视频和图片两种类型

### 11.3 数据一致性类缺陷示例
- 标题：切片已写入 PostgreSQL，但 Qdrant 中无对应 point
- 模块：素材/数据
- 数据影响面：PostgreSQL、Qdrant
- 应记录：
  - `slice_id`
  - `material_id`
  - Qdrant collection
  - 回调日志
- 回归要求：
  - 复测切片回调
  - 复测向量化回调
  - 复测检索接口

## 14. 遗留风险记录规则
1. 未关闭缺陷必须写清替代方案或规避方式。
2. 不阻断发布的缺陷也必须写清对演示的影响。
3. 若缺陷影响评委体验，需要在演示脚本中规避。
4. 若缺陷涉及真实数据缺失、Mock 限制、外部 API 波动，也必须单独注明。

## 15. 附录

### 15.1 编号规则
- Bug：`BUG-YYYYMMDD-序号`（编号前缀使用紧凑日期格式 `YYYYMMDD`）
- 回归：`REG-YYYYMMDD-序号`（同上）
- 展示时间戳（如执行时间）统一使用 ISO 8601 格式 `YYYY-MM-DD HH:MM`，与编号前缀的紧凑格式用途不同

### 15.2 关联规则
- 每个 Bug 必须至少关联一个测试用例或需求点
- 每个 P0 Bug 必须至少有一条回归记录
- 每个回归失败项必须重新进入台账并更新状态

### 15.3 与测试文档联动规则
- `整体测试用例文档.md` 中失败的 P0/P1 核心用例，必须进入本台账
- 台账关闭前，必须完成缺陷点回归、同模块关联回归，必要时完成主链路冒烟回归
