# TikStream AI 数据库设计文档

## 1. 文档目标与适用范围

### 1.1 文档名称
TikStream AI——电商场景 AIGC 带货视频生成系统数据库设计文档

### 1.2 文档目标
本文档用于统一 TikStream AI 的数据结构设计，明确关系型主数据、向量检索数据、对象存储引用与运行态任务数据之间的边界和协作关系，作为后端建模、Prisma Schema 编写、接口设计、前后端联调、Worker 协同和答辩复核的统一依据。

### 1.3 适用范围
本文档覆盖以下数据域：
- 商品与素材资产域
- 剧本与分镜结构化域
- 创作任务与分镜渲染域
- 模板与爆款拆解扩展域
- 向量检索、对象存储、缓存与队列相关的数据边界

### 1.4 设计口径
当前版本采用 PostgreSQL 16 作为关系型主存储，Qdrant 作为多模态向量检索存储，MinIO 作为对象存储，Redis 作为限流、信号量与 BullMQ 作业底座。数据库设计以竞赛 MVP 可落地、可演示、可扩展为原则，不引入复杂组织权限或超出当前阶段的商业化多租户模型。

## 2. 数据库设计原则与边界

### 2.1 设计原则
1. 主链路优先：优先服务“素材—剧本—创作—分析”主链路的数据稳定性。
2. 结构化优先：所有关键业务对象均需可结构化存储、检索、追踪与复用。
3. 媒体与元数据分离：大文件不直接落关系库，关系库只存对象引用、状态和检索线索。
4. 商品域隔离：素材检索、剧本生成、创作任务优先在单商品上下文内闭环，避免跨商品污染。
5. 状态可追踪：素材处理、切片处理、创作任务必须具备明确状态与阶段流转。
6. 可扩展：为模板市场、爆款结构化分析、归因看板和商业化演进保留数据位点。

### 2.2 当前阶段边界
本期数据库设计明确不覆盖以下能力：
- 多商家组织、部门、审批流和复杂权限模型
- 账单、结算、套餐、租户隔离等商业 SaaS 能力
- 真实投流回流广告平台对接与效果原始日志入库
- 重型因果推断数据仓库与离线数仓体系

### 2.3 数据边界说明
- PostgreSQL：保存商品、素材、切片、剧本、分镜、创作任务、分镜渲染结果、模板、爆款拆解结果等结构化业务数据。
- Qdrant：保存切片向量及 Payload，用于商品内语义召回和过滤检索。
- MinIO：保存原始素材、缩略图、切片流、字幕、音轨、渲染缓存、最终视频等对象文件。
- Redis：保存 Token Bucket 计数、并发信号量、BullMQ 队列运行态和分布式锁。
- DuckDB：用于分析看板预计算，离线执行留存曲线、归因热力图、桑基图等聚合查询，结果以 JSON 形态提供给 NestJS 分析接口消费。DuckDB 产生的预计算结果不写入 PostgreSQL，仅作为只读分析中间层存在。

## 3. 存储架构总览

### 3.1 存储分层
```text
业务应用层
  ├─ 商品/素材/剧本/创作 API
  ├─ GPU Slicing Worker
  ├─ Remotion Render Worker
  └─ 分析与模板服务
          │
          ├─ PostgreSQL 16
          │    ├─ products
          │    ├─ materials
          │    ├─ material_slices
          │    ├─ scripts
          │    ├─ script_shots
          │    ├─ creations
          │    ├─ shot_renders
          │    ├─ templates
          │    └─ viral_video_analyses
          │
          ├─ Qdrant
          │    └─ asset_slices (512 维向量 + Payload)
          │
          ├─ MinIO
          │    ├─ raw assets
          │    ├─ thumbnails
          │    ├─ slices
          │    ├─ subtitles
          │    ├─ audio tracks
          │    ├─ render cache
          │    └─ export videos
          │
          ├─ Redis / BullMQ
          │    ├─ rate limit counters
          │    ├─ distributed semaphores
          │    └─ queue jobs / locks
          │
          └─ DuckDB（分析预计算）
               ├─ retention curve aggregation
               ├─ attribution heatmap
               └─ sankey / ab-compare
```

### 3.2 设计理由
1. 向量库与关系库分离：避免在 PostgreSQL 中直接承担高维向量检索压力。
2. 对象存储与元数据分离：大体积媒体文件仅保存在 MinIO，通过 URL 或对象 Key 引用。
3. 运行态与主数据分离：BullMQ 的瞬时队列态与 PostgreSQL 的可持久业务态协同，但不混为一体。
4. 结构化数据主从统一：所有业务规则最终回落到 PostgreSQL 的主键、外键、唯一键、状态字段和 JSONB 承载结构。

## 4. 核心实体清单

### 4.1 主业务实体
1. Product：商品主实体，承载商品标题、SKU、类目、卖点摘要和统一上下文。
2. Material：素材主实体，承载上传的图片、视频、参考素材等基础元数据。
3. MaterialSlice：素材切片实体，承载视频切片、语义描述、切片时长与关键帧信息。
4. Script：剧本主实体，承载一条可执行剧本的整体结构、风格、生成模式与原始 JSON。
5. ScriptShot：分镜实体，承载镜头时长、画面描述、旁白、字幕、运镜和安全区坐标。
6. Creation：创作任务主实体，承载成片任务状态、进度、当前阶段、导出地址与失败原因。
7. ShotRender：单分镜渲染实体，承载增量缓存、渲染文件、命中切片和渲染状态。

### 4.2 扩展业务实体
1. Template：模板市场和策略模板实体，承载灵感模板、策略因子和适用范围。
2. ViralVideoAnalysis：爆款视频结构化拆解实体，承载来源链接、结构化分析结果和参考因子。

### 4.3 外部系统映射对象
1. Qdrant `asset_slices`：承载切片的 512 维向量及商品过滤 Payload。
2. MinIO Object：承载 `origin_url`、`thumbnail_url`、`stream_url`、`render_path`、`video_url` 等外部对象引用。
3. Redis/BullMQ：承载速率限制、并发信号量、任务锁与队列运行态。

## 5. ER 实体关系图

### 5.1 关系图
```text
Product (1)
  ├── (N) Material
  │         └── (N) MaterialSlice
  ├── (N) Script
  │         └── (N) ScriptShot
  ├── (N) Creation
  │         └── (N) ShotRender
  ├── (N) Template [可选关联，product_id 可为空，当前版本为全局预置]
  └── (N) ViralVideoAnalysis [可选关联，product_id 可为空]

Script (1) ─── (N) Creation [通过 script_id 关联，基于固定剧本发起成片]
ScriptShot (1) ─── (N) ShotRender [通过 script_shot_id 对应局部重渲染]
ScriptShot (N) ─── (1) MaterialSlice [可选关联，通过 selected_slice_id 对应 slice_id，用于人工指定或自动召回素材切片]
MaterialSlice (1) ─── (1) Qdrant asset_slices [通过 slice_id 对应]
```

### 5.2 关系说明
- 一个 Product 可以拥有多条素材、多条剧本和多个创作任务。
- 一个 Material 可以拆分为多个 MaterialSlice。
- 一个 Script 可以包含多个 ScriptShot。
- 一个 Creation 可以关联多个 ShotRender，用于分镜级增量渲染与缓存复用。
- Script 与 Creation 通过 `script_id` 建立关联关系，基于固定剧本发起成片。
- ShotRender 通过 `script_shot_id` 绑定具体分镜，支撑单分镜修改后的局部重渲染、失败回查和缓存复用。
- ScriptShot 可选记录 `selected_slice_id`，用于承接人工替换素材和自动召回素材两种模式。
- MaterialSlice 与 Qdrant 中的 Point 通过 `slice_id` 建立逻辑一一映射关系。

## 9. 数据约束与一致性规则

### 9.1 业务约束
1. 所有上传素材必须绑定有效 `product_id`。
2. 图片最大 10MB、视频最大 200MB 的校验在应用层完成，数据库侧要求保留 `file_size_bytes`、`mime_type` 和对象引用字段，承接校验结果与后续排查。
3. 成片总时长默认不超过 15 秒；该规则由服务层在保存 Script 与 ScriptShot 时完成聚合校验，数据库侧持久化 `video_duration` 及分镜明细结果。
4. 视频切片时长默认控制在 1.5s–4.0s，剧本单分镜时长需受 1.5s–5.0s 约束。
5. 每个分镜必须同时具备检索查询、可执行画面描述、运镜、转场、旁白、字幕和安全区字段。
6. 检索与创作优先在同一 `product_id` 范围内完成召回。
7. 任务阶段推进需幂等，重复推进同一阶段不得破坏最终任务状态。
8. 爆款视频仅保存结构化分析结果，不保存需复刻的原始版权视频内容。
9. 前端刷新后的任务恢复观察以 `task_id` 为主键，结合 `status`、`current_stage`、`progress`、`started_at`、`finished_at` 恢复当前视图。

### 9.2 关系约束
1. 删除 Product 时级联删除其 Materials、Scripts、Creations；Templates 和 ViralVideoAnalyses 如已沉淀为公共资产，可按业务选择级联删除或置空 `product_id`。
2. 删除 Material 时级联删除其 MaterialSlices。
3. 删除 Script 时级联删除 ScriptShots；若该 Script 已被 Creation 引用，则禁止直接删除，需先处理关联创作任务。
4. 删除 Creation 时级联删除其 ShotRenders。
5. ShotRender 必须绑定有效 `script_shot_id`，以确保局部重渲染结果能稳定回挂到分镜。
6. `shot_renders.script_shot_id` 采用级联删除是当前标准口径：当分镜被删除时，其对应的局部渲染产物应一并删除，避免遗留失去归属的缓存记录。

### 9.3 数据一致性约束
1. PostgreSQL 中 `slice_id` 与 Qdrant Point ID 必须保持一致。
2. PostgreSQL 中的对象引用字段必须指向已存在或可恢复的 MinIO 对象。
3. `progress` 范围固定在 0~100。
4. `script_id + shot_index` 组合必须唯一。
5. `sku_code`、`task_id`、`slice_id` 必须全局唯一。

## 10. 索引与查询设计

### 10.1 高频查询场景
1. 按商品查看素材列表、剧本列表、创作任务列表。
2. 按状态筛选素材处理结果与创作任务结果。
3. 按时间排序查看最近上传、最近创建、最近失败任务。
4. 按 `task_id`、`trace_id` 定位长任务。
5. 按 `script_id + shot_index` 读取分镜列表。

### 10.2 索引策略
- `products.sku_code`：唯一索引，用于商品唯一识别和跨模块对账
- `materials.product_id`、`materials.status`、`materials.created_at`：支撑素材工作台按商品、状态、时间分页查询
- `material_slices.material_id`、`material_slices.slice_id`、`material_slices.status`、`material_slices.duration`：支撑切片处理链路、状态筛选和向量映射定位
- `scripts.product_id`、`scripts.generation_mode`、`scripts.created_at`：支撑剧本历史查看与模式筛选
- `script_shots.script_id`、`script_shots.selected_slice_id`、`script_shots.compliance_status`、`script_shots(script_id, shot_index)` 唯一索引：支撑分镜定位、人工换素材与合规检查
- `creations.product_id`、`creations.status`、`creations.current_stage`、`creations.task_id`、`creations.trace_id`、`creations.started_at`、`creations.finished_at`：支撑任务页查询、恢复观察和链路排查
- `shot_renders.creation_id`、`shot_renders.script_shot_id`、`shot_renders.status`、`shot_renders.cache_hash`、`shot_renders.slice_id`：支撑局部重渲染、缓存命中和结果回查
- JSONB 字段（如 `raw_json`、`factor_json`、`report_json`）仅在存在结构化检索需求时使用 GIN 索引，避免无效索引膨胀

### 10.3 查询设计说明
- 关系库只负责结构化过滤与上下文定位，语义召回交由 Qdrant 完成。
- 素材检索流程通常先按 `product_id` 定位，再按结构化条件缩小范围，再调用向量库做相似检索。
- 创作任务页优先使用 `status + created_at` 组合查询，避免全表扫描。

## 11. 数据同步策略

### 11.1 PostgreSQL 与 Qdrant 同步
1. MaterialSlice 写入 PostgreSQL 成功后，异步触发向量化任务。
2. 向量生成成功后，将 `slice_id`、`product_id`、`duration`、`dense_caption` 写入 Qdrant Payload。
3. 若 Qdrant 写入失败，保留 PostgreSQL 主记录并进入重试队列。
4. 以 `slice_id` 作为幂等键，保证重复写入不产生脏数据。

### 11.2 PostgreSQL 与 MinIO 同步
1. 文件先落 MinIO，再写入 PostgreSQL 元数据记录。
2. 若对象上传失败，不应创建成功态业务记录。
3. 若元数据写入失败，应记录补偿事件并清理孤儿对象或转入人工排查。
4. `origin_url`、`stream_url`、`render_path`、`video_url` 仅保存对象引用，不保存媒体二进制。

### 11.3 PostgreSQL 与 Redis/BullMQ 协同
1. Creation 表保存业务可见任务状态；BullMQ 保存运行中作业态与锁。
2. 每次阶段推进时，同时更新队列事件与 Creation 的 `current_stage` / `progress`。
3. 页面恢复观察状态时，以 PostgreSQL 的任务主记录为准，再补拉 SSE / 队列运行态。
4. `status` 与 `current_stage` 分层维护：前者表达任务总状态，后者表达当前执行阶段，避免任务列表展示语义与状态机阶段语义混淆。
5. `FINISHED` / `FAILED` 在 `CreationStage` 中作为终态阶段枚举值，在 `CreationStatus` 中作为终态状态枚举值，两者语义不同但可同时出现（例：取消任务时 `status=CANCELED` 且 `current_stage` 记录最后执行到的阶段）。

## 12. 数据备份与恢复策略

### 12.1 备份策略
1. PostgreSQL 每日全量备份，关键演示期增加阶段性快照。
2. 关键表（products、materials、scripts、creations）保留最近多个版本备份。
3. MinIO 对重要目录（raw assets、render-cache、exports）做对象级定期同步或快照。
4. Qdrant 可定期导出 collection snapshot，保障切片向量可恢复。

### 12.2 恢复策略
1. 关系库优先恢复核心主表，再校验对象存储引用完整性。
2. 若 Qdrant 丢失，可基于 PostgreSQL 的切片主记录与对象引用重新生成向量。
3. 若 render-cache 丢失，不影响主链路正确性，但会增加重渲染成本。
4. 演示环境恢复目标以“主链路可运行、核心样例可演示”为第一优先级。

### 12.3 数据巡检
1. 定期检查 PostgreSQL 与 MinIO 之间的孤儿引用。
2. 定期检查 `slice_id` 是否在 Qdrant 中存在对应 Point。
3. 定期检查失败任务、异常状态切片和长时间未完成任务。

## 14. 命名规范与变更约定

### 14.1 命名规范
1. 表名统一采用复数蛇形命名，如 `material_slices`、`shot_renders`。
2. Prisma 模型名采用 PascalCase，如 `MaterialSlice`、`ShotRender`。
3. 数据库字段统一采用蛇形命名，通过 Prisma `@map` 映射到驼峰属性。
4. 枚举名统一采用业务语义英文命名，避免缩写不清。

### 14.2 变更约定
1. 所有表结构变更必须通过 Prisma Migration 管理。
2. 新增字段需同步更新：表结构设计、数据字典、枚举定义、Prisma Schema。
3. 关键状态枚举变更需同步更新后端状态机、前端状态展示与 SSE 事件文档。
4. 若新增扩展实体，应明确归属为“当前主设计”或“后续预留设计”，避免范围失控。
