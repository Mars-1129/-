# TikStream AI 代码规范与开发规约文档

## 1. 文档目标与适用范围

### 1.1 文档名称
TikStream AI——电商场景 AIGC 带货视频生成系统代码规范与开发规约文档

### 1.2 文档目标
本文档用于统一 TikStream AI 的代码风格、模块边界、分支协作、提交流程、接口变更、联调约定和交付门禁，确保两人协作下代码可维护、可审查、可回滚，并与全局 API、架构设计和 CI/CD 约束保持一致。

### 1.3 适用范围
本文档适用于：
- `apps/server-gateway` 后端实现
- `apps/web-client` 前端实现
- `workers/gpu-slicing-worker` 与 `workers/remotion-render-worker`
- `packages/shared-types` 共享类型
- `scripts/*` 预计算与种子脚本
- API、DTO、Service、Controller、Worker、Hook、Store、Type、配置、测试与文档变更

### 1.4 文档定位
本文档不重复描述业务需求，而是把 `文档.md`、`整体系统架构设计文档.md` 和 `.github/workflows/tks_production_ci_cd.yml` 中已明确的工程约束固化为可执行规约。所有开发、联调和提交流程都必须以本规约为准。

---

## 2. 代码与目录边界

### 2.1 Monorepo 边界
项目采用 Monorepo 组织方式，目录职责如下：
- `apps/web-client`：React / TypeScript 前端工作台
- `apps/server-gateway`：NestJS 核心网关与业务编排
- `workers/gpu-slicing-worker`：素材切片、向量化与按需 AI 处理
- `workers/remotion-render-worker`：Remotion 渲染、TTS、拼接与导出
- `packages/shared-types`：前后端共享类型与事件定义
- `scripts`：离线预计算、种子数据、辅助脚本

### 2.2 分层原则
1. UI 层只负责展示、交互与状态触发，不承载业务编排。
2. Controller 只做协议适配、校验与授权边界，不写重业务逻辑。
3. Service 层承载业务编排，Repository 只做数据访问。
4. Worker 只处理异步任务，不直接承担前端交互逻辑。
5. 共享类型只放稳定契约，不放临时实现。

### 2.3 命名规则
- 目录名与文件名统一使用 kebab-case（如 `material-slice`、`script-shot`）。
- 类名、接口名、枚举名使用 PascalCase（如 `MaterialService`、`CreationStatus`）。
- 变量名、函数名、方法名使用 camelCase（如 `getMaterialById`、`uploadFile`）。
- 数据库字段与 API 返回字段统一使用 snake_case（如 `product_id`、`created_at`）。
- 常量与枚举值使用 UPPER_SNAKE_CASE（如 `QUEUE_ALLOCATION`、`MAX_FILE_SIZE`）。
- Controller / Service / DTO / Module 使用清晰后缀。
- 前端组件、Hook、Store、Service 分层明确。
- 共享状态、任务状态、接口状态命名必须与 API 和数据库文档一致。

## 3. 分支与协作规约

### 3.1 分支策略
- `main`：稳定主线，承载可发布版本
- `develop`：日常集成分支，功能分支完成后发起 PR 合入
- 功能分支：按 `feat/<功能简述>` 或 `fix/<问题简述>` 创建，完成后发起 PR 合入 `develop`；`develop` 验证通过后合入 `main`

### 3.2 合入规则
1. 所有变更必须通过 PR。
2. 进入 `main` 前必须通过 Code Review。
3. 与 API 相关的变更必须同步更新 OpenAPI / 接口文档。
4. 影响运行时或部署的变更必须同步更新部署与运维文档。

### 3.3 双人协作原则
- 按模块分工，但接口契约必须先对齐。
- 任何跨模块字段变更都要同步共享类型与文档。
- 任务交接必须明确：输入、输出、状态、错误码、联调口径。

### 3.4 冲突处理原则
- 优先保留契约正确性，再处理局部实现差异。
- 不用“临时兼容字段”掩盖冲突。
- 数据模型、接口模型、状态枚举冲突时，以全局文档和设计文档为准。

## 4. 提交与审查规约

### 4.1 Commit 规约
- 每次提交只解决一个清晰问题。
- 提交信息必须包含目的而不是过程。
- API、部署、运维相关改动必须在提交中体现影响范围。

### 4.2 PR 规约
PR 必须包含：
- 变更摘要
- 影响范围
- 验证结果
- 接口/环境/运维是否变更
- 回滚方式

### 4.3 Review 重点
- 接口字段是否与全局 API 对齐
- 异步任务状态是否与创作链路一致
- 前端展示结构是否能直接消费后端结果
- 是否引入了与设计无关的多余抽象

---

## 5. 代码风格规约

### 5.1 TypeScript / JavaScript
- 使用明确类型，不依赖隐式推断遮蔽边界。
- DTO、Enum、常量、状态机使用显式命名。
- 不在业务逻辑中拼接魔法字符串。
- 共享字段名与数据库字段名保持一致或有明确映射。

### 5.2 前端规约
- 页面逻辑放在 view / hook / store / service 分层中。
- 交互组件优先复用 `shadcn/ui` 与 `Tailwind CSS` 的通用模式。
- 图表数据必须与 ECharts 的消费结构直接匹配。
- 长任务状态由统一的任务状态与 SSE 事件驱动。

### 5.3 后端规约
- Controller 只做协议层。
- Service 只做业务编排。
- Worker 只做任务执行。
- 错误响应必须统一结构化返回。

### 5.4 异步与状态规约
- 所有长任务必须有唯一追踪键。
- 状态流转必须显式。
- 失败原因必须可追踪、可恢复、可告警。

### 5.5 Agent / 工作流代码组织规约
本项目采用基于 NestJS 状态机编排的轻量自研工作流架构，代码组织需遵循以下规约：
- 状态机定义集中管理于 `services/state-machine/`，每个业务域（创作、素材）的状态流转为独立模块。
- BullMQ Job Processor 只负责调用 Service 方法，不在 Processor 中直接操作数据库或调用 AI 模型。
- Prompt 模板统一存放于 `services/prompts/` 目录，按业务场景命名，不在 Service 代码中硬编码 Prompt 文本。
- AI 模型调用统一通过 `services/ai/` 下的 Provider 封装，承担限流、重试、降级逻辑，Service 层不直接调用模型 SDK。
- 工作流上下文（`task_id`、`trace_id`、`product_id`、阶段信息）通过 NestJS CLS（AsyncLocalStorage）贯穿整个请求/任务生命周期。

## 6. API 与契约变更规约

### 6.1 OpenAPI 优先
- 新增或修改接口必须先更新 OpenAPI / 接口文档。
- CI 中 `pnpm lint` 随 P2 阶段逐步启用，`pnpm openapi:lint` 为已启用硬门禁。
- 接口实现不得先于契约失控上线。

### 6.2 共享类型规约
- API 请求 / 响应 DTO 优先放入共享类型层或明确对应模块。
- 前后端字段名、状态枚举、错误码必须同源。

### 6.3 破坏性变更规约
- 破坏性字段修改必须提前评估影响。
- 不以临时兼容字段绕过契约升级。
- 影响部署或数据迁移的变更必须同步环境与运维文档。

---

## 7. 质量门禁

### 7.1 必做检查
在进入主线前至少满足以下门禁：
- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm build`

### 7.2 本地检查顺序
建议顺序：
1. 安装依赖
2. 运行类型检查
3. 运行单测
4. 检查接口文档
5. 再做集成验证

### 7.3 禁止项
- 不通过验证直接合入主线
- 不同步文档就更改对外契约
- 不清楚依赖影响就修改共享类型
- 用复杂抽象替代清晰的模块边界

---

## 8. 双人协作落地规则

### 8.1 任务切分
- 人员 A：素材、创作、前端工作台、渲染相关
- 人员 B：剧本、分析看板、接口契约、数据分析相关

### 8.2 交接要求
交接时必须提供：
- 当前分支
- 已完成接口
- 待确认字段
- 需要联调的环境变量
- 验证方法

### 8.3 联调顺序
- 先契约，后实现
- 先共享类型，后业务代码
- 先 mock，后联调
- 先本地验证，再进入部署验证

---

## 9. 验证标准

1. 代码风格是否符合目录与分层边界。
2. API 变更是否同步 OpenAPI 与接口文档。
3. PR 是否包含影响范围、验证结果和回滚方式。
4. 质量门禁是否覆盖 lint、typecheck、test、build。
5. 双人协作是否保持模块边界清晰、共享类型一致。
