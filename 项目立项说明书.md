# TikStream AI 项目立项说明书

## 1. 项目概述

### 1.1 项目名称
TikStream AI——电商场景 AIGC 带货视频生成系统

### 1.2 项目定位
TikStream AI 是面向 TikTok Shop 等国际电商平台的 AIGC 带货视频生产系统，目标是在竞赛周期与有限算力条件下，构建一套可运行、可演示、可扩展的一站式短视频生成解决方案。系统围绕“素材入库—剧本生成—智能创作—效果分析”的完整链路，帮助跨境电商商家低成本、高效率地生成符合平台规范的 15 秒竖屏带货短视频。

### 1.3 立项目的
本项目立项的直接目的，是围绕比赛要求交付一套具备完整业务闭环、明确工程落地路径、可进行现场答辩与演示的准生产级 MVP 系统。项目通过整合多模态素材处理、大模型剧本生成、自动化视频合成、Mock 驱动的数据归因分析等能力，证明团队具备从业务抽象到系统设计、从需求收敛到工程交付的完整能力。

## 2. 项目背景与立项依据

### 2.1 行业背景
跨境电商短视频营销已成为 TikTok Shop 等平台的重要增长手段，但大量中小商家缺少专业摄制团队、海外拍摄能力与视频剪辑人员，往往只有商品链接、若干产品图或工厂实拍素材，难以高质量、高频率地产出符合投流需求的带货视频内容。

### 2.2 业务痛点
当前目标用户普遍面临以下问题：

1. 素材能力不足：只有少量产品图、白底图或横屏厂拍视频，缺乏可直接投流的短视频素材。
2. 内容生产成本高：脚本策划、剪辑、配音、字幕与导出需要多人协作，时间成本和人力成本较高。
3. 爆款经验难以沉淀：爆款视频的镜头结构、文案节奏、视觉风格与音频策略缺乏可复用的数据化表达。
4. 批量投放难以支撑：需要快速生产多风格、多语言、多版本素材用于 A/B 测试与持续投流。
5. 技术落地门槛高：传统视频生成方案对 GPU 显存、模型部署和系统并发控制要求高，竞赛周期内容易失控。

### 2.3 立项价值
本项目的核心价值体现在以下几个方面：

- 降低门槛：商家仅需上传商品素材或输入商品卖点，即可自动生成可投放视频。
- 提升效率：通过素材结构化、机器可执行剧本、增量渲染和四轨合成，缩短视频生产周期。
- 强化复用：通过模板、爆款仿写、策略因子库沉淀，提升内容方法论可复用性。
- 体现工程能力：在有限 API 配额和本地显存预算下，完成系统性资源治理与全链路交付。
- 支撑答辩与扩展：竞赛阶段交付 MVP，后续具备商业化与规模化演进空间。

## 3. 项目目标

### 3.1 总体目标
构建一套覆盖素材处理、剧本生成、视频创作、结果分析的完整 AIGC 带货视频生产系统，在竞赛约束下实现全链路打通、核心功能可用、架构方案可信、工程实现可验证。

### 3.2 分阶段目标

#### P0 核心必交付目标
1. 支持商品图片/视频素材上传与入库。
2. 支持素材切片与 Dense Caption 语义打标。
3. 支持基于商品卖点的一键剧本生成。
4. 支持基础分镜结构化输出与分镜编辑。
5. 支持一键成片、任务进度跟踪与最终导出。
6. 形成可演示的完整主链路闭环。

#### P1 增强交付目标
1. 支持多模态向量检索与混合检索。
2. 支持多种剧本生成模式、分镜级局部编辑与配时校验。
3. 支持四轨合成（P1）、局部增量重渲染、任务恢复和失败重试。
4. 支持分析看板、历史任务、模板市场等扩展能力。

#### P2 亮点与加分目标
1. 支持多因子归因分析、A/B 对比与一键自愈。
2. 支持 API 配额治理、VRAM 预算治理、可观测性与 CI/CD。
3. 体现 AI Agents、增长飞轮、长期商业化扩展等创新价值。

## 4. 目标用户与使用场景

### 4.1 目标用户

#### 1）跨境电商中小商家
特点：素材少、预算有限、没有专业视频团队。
需求：快速把商品图转化为可直接投放的短视频。

#### 2）品牌跨境运营人员
特点：需要批量投流、频繁测试不同风格素材。
需求：批量生成多版本视频，支撑 A/B 测试。

#### 3）内容营销与剪辑团队
特点：有内容经验，但难以沉淀爆款方法论。
需求：结构化保存并复用优秀脚本、镜头与风格策略。

#### 4）工厂型或档口型卖家
特点：手中多为横屏厂拍素材，缺少精剪视频。
需求：对既有素材做切片、打标、检索与自动混剪。

### 4.2 核心使用场景
1. 上传商品图或视频素材，自动完成切片、打标和入库。
2. 输入商品卖点，快速生成符合带货逻辑的 15 秒短视频脚本。
3. 在分镜编辑器中微调台词、镜头、切片与字幕内容。
4. 一键生成成品视频，并在前端实时查看任务进度与结果。
5. 通过看板观察效果指标与归因结果，反向优化下一版素材。

## 5. 项目范围定义

### 5.1 本期建设范围
本项目围绕四个核心模块开展建设：

1. 素材模块：负责素材上传、切片、Dense Caption、向量化、检索与基础资产管理。
2. 剧本模块：负责脚本生成、爆款仿写、模板组装、分镜结构化与配时校验。
3. 创作模块：负责素材匹配、图生视频兜底、配音、字幕、BGM、渲染与导出，默认输出 1080×1920 / 30fps / H.264 标准视频。
4. 分析看板模块：负责基于 Mock 数据的归因分析、留存曲线、A/B 对比与策略反馈。

### 5.2 明确不做范围
为控制竞赛周期内的工作边界，本期明确不纳入以下能力：

1. 不自建重型本地视频生成大模型训练与微调体系。
2. 不建设真实线上广告投放平台对接与真实回流数据采集链路。
3. 不支持复杂多人协同编辑、审批流、组织级权限中心。
4. 不支持完整商业 SaaS 账单、结算、计费与租户管理能力。
5. 不实现需要高额算力与高风险部署的跨帧重型追踪、实时数字人生成、复杂因果推断服务。

### 5.3 技术选型边界
在满足赛题主链路与交付要求的前提下，项目技术选型遵循“核心主栈建议统一、专项能力允许自由选择”的原则：
- 前后端主栈优先采用 React、Node.js、TypeScript 与火山引擎 OpenAPI，保证与赛题预期技术路线一致。
- UI 组件库、图表库、Agent 框架、向量库与补充开源模型可结合工程需要自由选型，但不得影响主链路打通、文档可复核性与现场演示稳定性。
- 允许使用 AI 编程工具辅助研发，但工具本身不作为系统运行依赖，也不替代对架构质量、代码质量与交付质量的把控。

### 5.4 范围控制原则
- 保证主链路完整性优先于局部炫技。
- 保证系统稳定可演示优先于极致模型复杂度。
- 保证工程可信性优先于堆砌不可落地功能。

## 6. 功能优先级划分

### 6.1 P0 核心功能
- 商品素材上传与基础管理
- 素材切片与 Dense Caption 语义打标
- 一键剧本生成
- 分镜级结构化脚本输出
- 一键成片与视频导出
- 基础音视频轨合成（视频轨+旁白轨）
- 图生视频基础兜底
- 长任务进度追踪

### 6.2 P1 增强功能
- 向量检索与混合检索
- 爆款仿写、灵感模板
- JSON Patch 分镜编辑
- 音节配时实时校验
- BGM、字幕轨合成（四轨完整版，视频轨+旁白轨已在P0）
- 即时预览
- 任务历史、断点恢复、失败重试
- 模板市场与看板基础能力
- 分镜留存曲线

### 6.3 P2 加分功能
- 高还原度即时预览（与最终导出画质一致）
- 多因子归因热力图
- 视听留存桑基图
- A/B 对比
- 一键自愈
- API 配额治理
- VRAM 预算治理
- CI/CD、可观测性与演进蓝图

## 7. 双人协作分工与职责边界

### 7.1 人员 A 负责范围
人员 A 负责素材模块与创作模块，具体包括：
- 素材上传、切片、打标、向量化与检索链路
- 视频创作、局部增量渲染、导出与音视频合成链路
- 与对象存储、Qdrant、GPU Worker、Render Worker 相关的实现与联调

### 7.2 人员 B 负责范围
人员 B 负责剧本模块与分析看板模块，具体包括：
- Prompt 驱动剧本生成、爆款仿写、模板组装与剧本 Schema 管理
- 分镜编辑、配时校验、合规词法拦截等逻辑
- 数据看板、A/B 对比、归因分析与增长反馈链路

### 7.3 协作接口边界
两人通过以下边界协同：
- 统一数据结构：商品、素材、切片、剧本、分镜、创作任务等核心实体保持一致定义。
- 统一接口契约：模块之间通过明确的 API、SSE 事件和任务状态流转衔接。
- 统一阶段目标：按 P0/P1/P2 分阶段集成，先打通主流程，再补强增强能力与亮点能力。

## 8. 资源清单与前置条件

### 8.1 技术资源
1. 前端运行环境：React 18 + TypeScript 5 + Vite 5
2. 前端组件体系：shadcn/ui + Tailwind CSS，专项能力组件使用 Remotion 4 Player 与 ECharts 5

## 10. 主要风险与应对策略

### 10.1 资源约束风险
风险：云端 API 配额有限、本地 VRAM 容量有限，容易出现限流或 OOM。
应对：采用 Token Bucket 限流、分布式信号量、模型常驻与按需释放、缓存优先与降级策略。

### 10.2 周期过短风险
风险：竞赛周期内功能点多，容易导致范围失控。
应对：严格按 P0/P1/P2 分级，先保主链路，再补亮点。

### 10.3 集成复杂度风险
风险：素材处理、脚本生成、视频创作、数据看板涉及多模块、多中间件和多模型。
应对：统一契约、分层设计、双人分工明确、逐阶段联调。

### 10.4 演示稳定性风险
风险：现场演示时可能受外部 API 波动、任务耗时、缓存状态影响。
应对：提前准备缓存样本、任务追踪、容错降级链路与健康检查机制。

### 10.5 真实数据不足风险
风险：难以获得真实投流回流数据。
应对：根据赛题允许范围采用 Mock 数据与 DuckDB 预计算方案，重点展示结构设计和分析逻辑。

在完成竞赛交付版 MVP 的基础上，项目后续可向以下方向演进：

1. 接入更强的本地或云端视频生成能力，进一步降低生成成本。
