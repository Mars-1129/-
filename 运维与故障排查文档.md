# TikStream AI 运维与故障排查文档

## 1. 文档目标与适用范围

### 1.1 文档名称
TikStream AI——电商场景 AIGC 带货视频生成系统运维与故障排查文档

### 1.2 文档目标
本文档用于沉淀 TikStream AI 在开发、演示、联调与准生产阶段的日常运维流程、健康检查、监控指标、故障定位方法、恢复步骤和回滚策略，确保系统发生异常时可以快速止损、准确定位、可控恢复。

### 1.3 适用范围
本文档覆盖：
- 日常巡检
- 健康检查
- 任务与队列排障
- 数据库、缓存、消息队列排障
- 镜像、发布、容器编排排障
- 迁移、导出、渲染、模型调用异常处理
- 服务重启、版本回滚、数据恢复
- AI 服务异常处理

### 1.4 文档定位
本文档基于 `文档.md`、`整体系统架构设计文档.md` 和 `.github/workflows/tks_production_ci_cd.yml` 中明确的可观测性、容错、部署和验证口径编写，是项目异常处理与恢复执行的统一手册。

- `task_id`
- `trace_id`
- `creation_id`
- `material_id`
- `slice_id`

## 3. 日常运维流程

### 3.1 每日巡检
1. 检查健康检查端点。
2. 检查数据库、Redis、Qdrant、MinIO 可用性。
3. 检查队列是否积压。
4. 检查任务成功率与失败率。
5. 检查 GPU / 内存水位。
6. 检查最近发布是否成功。

### 3.2 发布后巡检
1. 确认 Docker Compose 服务全部启动正常。
2. 确认 `/health` 正常。
3. 确认 Prisma migration 正常。
4. 走一遍上传、生成、导出、看板查询的烟测链路。

### 3.3 故障处置顺序
建议按以下顺序处理：
1. 确认影响范围
2. 判断是否为发布引起
3. 检查依赖服务
4. 检查任务状态与日志
5. 判断是否重启、回滚或降级
6. 记录复盘项

## 4. 监控指标

### 4.1 核心指标
- 任务成功率
- 平均生成耗时
- 队列堆积长度
- 模型调用失败率
- 缓存命中率
- GPU 水位
- 内存水位

### 4.2 任务级指标
- 创建成功率
- 分镜渲染成功率
- 导出成功率
- SSE 连接恢复率
- 局部重渲染成功率

### 4.3 依赖级指标
- PostgreSQL 连接数与响应
- Redis 可用性与延迟
- Kafka 消费滞后（仅在启用完整事件驱动架构时需要监控；MVP 阶段可降级为 BullMQ）
- Qdrant 检索状态
- MinIO 写入状态

## 5. 日志与追踪

### 5.1 日志分层
日志至少分为：
- 网关请求日志
- 任务编排日志
- Worker 执行日志
- 渲染日志
- 外部模型调用日志

### 5.2 关键字段
每条关键日志应包含：
- `task_id`
- `trace_id`
- 业务对象 ID
- 阶段状态
- 错误码
- 错误信息
- 时间戳

### 5.3 排障原则
- 先用 `task_id` 找链路，再用 `trace_id` 找请求。
- 先看阶段，再看依赖，再看外部模型。
- 先定位是否可重试，再决定是否回滚。

## 6. 常见故障与处理

### 6.1 健康检查失败
**现象**：`GET /health` 失败。  
**排查**：
1. 检查网关进程。
2. 检查数据库和缓存连接。
3. 检查依赖服务是否启动。
4. 检查最近发布是否引入配置错误。

**处理**：
- 先恢复依赖。
- 必要时重启网关。
- 如果是发布问题，回滚到上一稳定版本。

### 6.2 Prisma 迁移失败
**现象**：部署后迁移状态异常。  
**排查**：
1. 检查迁移脚本是否与当前 schema 冲突。
2. 检查数据库连接权限。
3. 检查是否有重复迁移或顺序冲突。

**处理**：
- 停止继续发布。
- 先修复迁移或回滚数据库变更。
- 保证应用版本与 schema 兼容。

### 6.3 队列积压
**现象**：任务增长但执行缓慢。  
**排查**：
1. 看 BullMQ/任务队列长度。
2. 看 Worker 是否存活。
3. 看 Redis 是否正常。
4. 看 GPU / 资源是否达到瓶颈。

**处理**：
- 检查 Worker 进程是否健康，必要时重启。
- 降低并发参数，减轻单 Worker 负载。
- 对非关键任务延迟重试。

### 6.4 模型调用限流或失败
**现象**：文本或视频模型返回失败。  
**排查**：
1. 检查限流参数是否达到上限。
2. 检查外部模型服务连通性。
3. 检查是否触发降级逻辑。

**处理**：
- 走重试。
- 进入降级提示词或兜底方案。
- 降低并发。

### 6.5 渲染失败
**现象**：分镜渲染或导出失败。  
**排查**：
1. 检查 Worker 是否运行。
2. 检查 FFmpeg / Chromium 依赖。
3. 检查对象存储写入。
4. 检查素材切片与字幕轨是否缺失。

**处理**：
- 定位到具体分镜。
- 尝试局部重渲染。
- 必要时回退到上一个可用版本。

### 6.6 对象存储异常
**现象**：上传、切片或导出写入失败。  
**排查**：
1. 检查 MinIO 连接与权限。
2. 检查 bucket 是否存在。
3. 检查磁盘空间。

**处理**：
- 恢复存储服务。
- 重新执行失败任务。

## 7. 恢复与回滚

### 7.1 服务重启
按依赖顺序重启：
1. 依赖服务
2. 网关
3. Worker
4. 前端

### 7.2 版本回滚
- 回滚 Git 提交或镜像版本。
- 重新执行 Docker Compose 启动。
- 回滚后再次执行 `/health` 与迁移检查。

### 7.3 数据恢复
- 恢复前先确认任务状态。
- 优先保证数据库与任务状态一致。
- 必要时先恢复核心业务表，再恢复派生结果。

### 7.4 AI 服务异常处理
- 先降低并发。
- 再启用降级策略。
- 再决定是否暂停相关任务队列。

---

## 8. 告警与升级

### 8.1 需要告警的情况
- 健康检查失败
- 队列持续积压
- 任务失败率持续升高
- GPU / 内存水位异常
- 外部模型调用失败持续出现
- 迁移失败

### 8.2 升级原则
- 影响主流程优先升级。
- 影响演示链路优先升级。
- 影响数据一致性必须立即升级。

---

## 9. 验证标准

1. 是否能通过 `/health` 和迁移检查快速判断服务状态。
2. 是否能通过 `task_id` / `trace_id` 完成链路定位。
3. 是否能覆盖依赖、队列、渲染、模型、存储等常见故障。
4. 是否给出明确的重启、回滚、恢复步骤。
5. 是否可直接用于日常运维与异常止损。
