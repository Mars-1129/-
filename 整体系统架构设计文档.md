# TikStream AI 整体系统架构设计文档

## 1. 文档概述

### 1.1 文档名称
TikStream AI——电商场景 AIGC 带货视频生成系统整体系统架构设计文档
  
### 1.2 文档目标
本文档用于定义 TikStream AI 的总体技术方案，统一系统分层、模块职责、技术选型、数据流转、非功能设计、资源治理与风险规避策略，作为研发实现、联调、运维和答辩的统一技术依据。

### 1.3 文档设计原则
- 保证竞赛版可跑通、可演示、可答辩。
- 保证在 API 配额与本地 VRAM 受限条件下的稳定运行。
- 保证模块职责清晰，便于双人协作开发与集成。
- 保留向商业化扩展的演进空间。

## 2. 架构设计目标与约束

### 2.1 架构目标
1. 打通“素材—剧本—创作—分析”全链路。
2. 将业务需求落地为可运行的工程化系统。
3. 支持异步任务、素材结构化、多模态检索与视频生成。
4. 支持长任务可追踪、失败可恢复、资源可治理。
5. 在竞赛答辩场景下提供可信的系统拓扑、限流策略、缓存策略与交付路径。

### 2.2 关键约束
1. 火山方舟 API 配额有限，需要严格控制 RPM、TPM 和并发。
2. 本地 GPU 显存有限，需要避免模型堆叠导致 OOM。
3. 比赛周期有限，需要优先实现主链路而非重型自研模型。
4. 数据看板允许 Mock 数据，因此重点在结构、逻辑和可视化，不在真实线上回流接入。

## 3. 总体架构概览

### 3.1 总体架构思想
系统采用前后端分离、事件驱动、异步任务调度与前后端同构混合渲染的总体架构。通过 NestJS 网关统一承接业务 API 与状态编排，通过 BullMQ 投递素材处理事件（竞赛 MVP 阶段；Kafka 保留为生产扩展选项），通过 Redis + BullMQ 调度长任务，通过 PostgreSQL、Qdrant 与 MinIO 分别承载结构化数据、向量检索与媒体对象存储。

### 3.2 总体架构拓扑
```text
前端浏览器（React / TypeScript）
  ├─ 素材管理工作台
  ├─ 智能剧本编辑器
  ├─ 创作与预览工作台
  └─ 分析看板
          │
          │ REST / SSE
          ▼
NestJS 核心网关 / 状态编排层
  ├─ Material Service
  ├─ Script Service
  ├─ Creation Service
  ├─ Token Bucket 限流中间件
  └─ Trace / 异常 / 配置模块
          │
          ├─ Kafka 事件总线
          ├─ Redis + BullMQ 长任务调度
          ├─ 火山方舟大模型接口
          └─ 存储与检索层
                  ├─ PostgreSQL
                  ├─ Qdrant
                  └─ MinIO
          │
          ├─ GPU Slicing Worker
          └─ Remotion Render Worker
```

## 4. 分层架构设计

### 4.1 前端层
职责：承接用户交互、上传、分镜编辑、实时预览、长任务状态展示与分析看板可视化。

核心能力：
- React SPA 页面组织
- Zustand 状态管理
- Remotion Player 前端预览
- ECharts 数据可视化
- SSE 长连接状态接收
- LocalStorage 长任务恢复

### 4.2 服务层
职责：统一 API 入口、业务校验、模块编排、资源治理、状态流转与错误处理。

核心能力：
- NestJS 模块化组织
- DTO / Schema 校验
- 限流、信号量与失败重试
- 任务状态机管理
- 对外接口与对内回调接口管理

### 4.3 数据层
职责：承载结构化业务数据、向量检索数据、对象存储数据与缓存元数据。

核心能力：
- PostgreSQL：商品、素材、切片、剧本、分镜、创作任务等关系数据
- Qdrant：ImageBind 向量存储与 Payload 过滤检索
- MinIO：原始素材、切片流、音轨、字幕、缓存、导出视频
- Redis：限流计数、分布式信号量、BullMQ 队列底座

### 4.4 AI 能力层
职责：提供文本生成、图生视频、多模态编码、TTS、ASR、SFX 提取和 ROI 检测等能力。

当前版本的 Agent / 工作流编排框架采用基于 NestJS 的轻量自研状态机架构，由 NestJS 服务层负责状态流转、Prompt 工作流组织、任务编排与异常治理，并结合 BullMQ、Redis、Kafka、SSE 与 Trace ID 形成可追踪、可降级、可恢复的执行链路。该方案优先满足竞赛 MVP 的工程落地需求，不额外引入复杂的外部多 Agent runtime。

核心能力：
- 云端模型：Doubao-Seed-2.0-pro、Doubao-Seedance-1.5-pro
- 本地常驻模型：Meta ImageBind、F5-TTS
- 本地按需模型：YOLOv11、HTDemucs、Faster-Whisper

### 4.5 接口与协作层
职责：定义 API 契约、回调协议、SSE 事件和任务状态流转，作为双人开发协同边界。

## 6. 关键业务流程与数据流

### 6.1 素材入库流程
```text
前端上传素材
  → 网关校验文件与商品绑定
  → MinIO 存储原始文件
  → 派发 Kafka 事件 asset.ingest.raw
  → GPU Worker 切片
  → 关键帧聚合并调用 Doubao 多模态能力（视觉+文本）生成 Dense Caption
  → 调用 ImageBind 生成 512 维向量
  → PostgreSQL 写入 material / material_slice
  → Qdrant 写入向量与 Payload
```

### 6.2 剧本生成流程
```text
用户输入商品卖点 / 选择模板 / 指定爆款
  → Script Service 组织 Prompt 与上下文
  → 调用 Doubao-Seed-2.0-pro
  → 返回结构化分镜 JSON
  → 后端执行 Schema 校验
  → 持久化 script / script_shot
  → 返回前端编辑器
```

### 6.3 视频创作流程
```text
用户点击一键成片
  → 创建 BullMQ 任务
  → Creation Service 进入状态机
  → 根据分镜检索切片
  → 命中缓存则直接使用
  → 未命中则调用 Seedance 图生视频补足
  → 生成 TTS 音频与词级时间戳
  → 生成字幕和 BGM 编排
  → 单分镜渲染与缓存
  → FFmpeg 分镜拼接（拼接后经 loudnorm 重编码完成响度标准化）
  → loudnorm 响度固化
  → MinIO 落盘导出视频
  → 返回 video_url
```

> **分镜数约束**：在 15 秒总时长约束下，结合单分镜 1.5s–5.0s 的硬性限制，理论上每段视频的分镜数范围为 3~10 个。推荐 4 分镜结构（Hook → Pain Point → Product Demo → CTA）作为默认叙事节奏（仅为内容策略建议，非系统硬性约束），此时各分镜平均时长建议控制在 3.0s–3.75s，避免单分镜过长导致总时长超限。

### 6.4 分析回流流程
```text
Mock 数据 / 预计算 JSON
  → NestJS 分析接口读取
  → 前端 ECharts 渲染
  → 用户观察 A/B 或归因结果
  → 触发一键自愈诊断
  → dry_run=true：返回建议摘要与受影响分镜
  → dry_run=false：复用 Creation Service / BullMQ 创建正式任务并进入 QUEUE_ALLOCATION
  → 生成 healed_creation_id 并继续下一轮剧本 / 分镜优化
```

### 7.5 选型原则
- 优先选择可快速集成、工程成熟的组件。
- 优先控制资源与复杂度，不引入重型难落地方案。
- 优先保留系统弹性和后续替换空间。

### 7.6 关键框架选型结论
- Agent / 工作流框架：当前版本采用基于 NestJS 状态机编排的轻量自研工作流架构，结合 BullMQ、Redis、Kafka 与 SSE 完成长任务编排、状态推进和异常治理。
- UI 组件库：当前版本采用 shadcn/ui 作为通用交互组件库，采用 Tailwind CSS 作为响应式布局与样式基础设施。
- 专项前端能力：Remotion 4 Player 负责视频预览与时间轴渲染，ECharts 5 负责分析看板可视化。
- 选择 shadcn/ui 的原因是其基于 Radix UI 无头组件与 Tailwind CSS 原子样式，适合 B 端工作台、时间轴编辑器和高定制交互页面，集成快、可控性强、后续可替换成本低。
- 当前不将 LangGraph / Claude Code SDK 写为主栈，是因为竞赛 MVP 更强调状态可控、链路可观测和资源约束下的稳定交付；LangGraph + Claude Code SDK 保留在后续商业化演进阶段。
- LangGraph、Claude Code SDK 等开源 Agent 运行时属于允许接入的可选路线，但在当前版本中不作为唯一主方案，避免在竞赛周期内引入不必要的运行时复杂度。

## 10. 性能、并发与资源治理设计

### 10.1 API 配额治理
- Doubao 文本模型请求采用 Token Bucket 控制在 80 RPM。
- 文本任务采用 5 帧聚合，降低调用次数。
- 请求过载时降级为压缩 Prompt 模式。

### 10.2 并发治理
- Seedance 采用分布式信号量限制并发，上限由环境变量 `SEEDANCE_MAX_CONCURRENCY` 控制（默认 3），基于 Redis 实现跨 Worker 协调。
- BullMQ 队列控制长任务消费节奏。
- 素材切片和渲染任务分离，避免互相阻塞。

### 10.3 VRAM 预算治理
常驻模型预算：
- Decord + TransNetV2：约 2.0GB
- Meta ImageBind：约 3.0GB
- F5-TTS：约 3.5GB
- 总计约 8.5GB

按需互斥模型预算：
- HTDemucs：约 2.0GB
- Faster-Whisper：约 2.0GB
- YOLOv11：约 1.5GB

设计结论：
- 保留约 7.0GB 安全余量给 FFmpeg、Chromium 和多分镜渲染时的显存峰值波动。
- 所有按需模型执行后立即释放显存，避免 OOM。
- 渲染多分镜时优先串行执行，控制 Chromium 实例数量，防止显存叠加溢出。

### 10.4 缓存治理
- 使用 SceneHash 对单分镜结果进行缓存。
- 素材切片优先检索缓存结果，减少图生视频调用。
- 导出前只重算变更分镜，减少整体渲染压力。

### 10.5 响度标准化
- 最终成片音频通过 FFmpeg loudnorm 滤镜进行响度标准化。
- 目标参数：平均集成响度 `-14.0 LUFS`，最大真峰值不高于 `-1.0 dBTP`。
- 标准化在 `LOUDNORM_COMPLIANCE` 阶段执行，保证全片音频响度统一。

## 11. 安全、容错与可观测性设计

### 11.1 安全设计
- 对上传文件做格式、大小、商品归属校验。
- 对 API 调用做限流与并发保护。
- 对外部模型接口异常做重试与降级。
- 对脚本文本做合规词法检测。
- 对引用的公开素材、参考视频与拆解结果保留来源声明与结构化分析留痕，避免形成无法追溯的内容资产。
- 所有敏感凭据均通过环境变量或密钥管理机制注入，不进入前端代码、公开仓库、日志与正式答辩材料。
- 生产与演示环境遵循最小权限原则，避免对象存储、数据库与模型调用权限过度暴露。

### 11.2 容错降级设计
- 文本模型限流后进入重试或 Prompt 降级。
- 视频模型并发满时进入延迟队列错峰。
- 素材检索失败时走图生视频兜底。
- 前端任务中断后可通过 task_id 自动恢复观察。

### 11.3 可观测性设计
- 所有创作任务生成全局唯一 task_id。
- task_id 贯穿网关、Kafka、Worker、渲染和导出链路。
- 对外暴露健康检查与资源统计接口。
- 长任务通过 SSE 派发标准阶段事件。
- 日志按网关请求、任务编排、Worker 执行、渲染输出、外部模型调用五类分层记录，便于问题定位。
- 关键监控指标至少覆盖任务成功率、平均生成耗时、队列堆积长度、模型调用失败率、缓存命中率与 GPU / 内存水位。
- 对失败率异常、任务长时间卡住、队列积压、资源水位过高等场景配置告警，支撑演示期和联调期的快速止损。

### 11.4 状态机阶段
- QUEUE_ALLOCATION
- ASSET_MATCHING
- AI_VIDEO_GENERATING
- TTS_GENERATING
- FFMPEG_STITCHING
- LOUDNORM_COMPLIANCE
- FINISHED / FAILED

## 14. 技术风险、预研结论与规避方案

### 14.1 重型本地视频模型风险
结论：不采用本地重型视频生成模型训练或运行方案。
规避：统一改为云端视频生成 API + 本地缓存优先策略。

### 14.2 显存溢出风险
结论：显存是竞赛环境下的最大物理风险。
规避：模型分常驻与按需两类，执行后立即释放，严控预算水位。

### 14.3 图数据库与复杂因果推断风险
结论：引入图数据库和重型因果推断框架 ROI 低、交付风险高。
规避：策略因子库落 PostgreSQL JSONB；看板基于 DuckDB 预计算。

### 14.4 演示不稳定风险
结论：竞赛现场对系统稳定性要求高于局部复杂度。
规避：加强缓存、状态机、健康检查、Trace 和降级策略。

## 15. 后续商业化演进方向

1. 引入更多区域与语言的文化对齐能力。
2. 引入更低成本的视频生成替代方案，降低云端调用成本。
3. 接入真实投流数据，升级归因与增长分析链路。
4. 引入多 Agent 协作运行时，增强自动诊断与自动优化能力。
5. 扩展为标准化商家 SaaS 平台，增加组织、权限、模板市场与运营能力。
