# TikStream AI 环境搭建与部署文档

## 1. 文档目标与适用范围

### 1.1 文档名称
TikStream AI——电商场景 AIGC 带货视频生成系统环境搭建与部署文档

### 1.3 适用范围
本文档覆盖：
- 本地开发环境搭建
- 依赖服务启动与配置
- Monorepo 依赖安装
- 前后端与 Worker 启动
- Docker Compose / 容器化部署
- 镜像构建与推送
- 部署后验证与回滚
- 环境变量与基础资源配置

### 1.4 文档定位
本文档基于 `文档.md`、`整体系统架构设计文档.md` 和 `.github/workflows/tks_production_ci_cd.yml` 中已明确的技术栈与流水线约束编写，是本项目环境、部署与发布的唯一执行参考之一。

## 3. 本地开发环境搭建

### 3.1 依赖安装
1. 安装 Node.js 22。
2. 安装 pnpm 9。
3. 执行依赖安装：优先使用 lockfile。
4. 确保本地具备 Docker 与 Docker Compose。

### 3.2 依赖服务启动
本地开发至少需要以下服务：
- PostgreSQL
- Redis
- Qdrant
- MinIO

建议先启动依赖服务，再启动业务服务与 Worker。

> 注意：Kafka 在竞赛 MVP 阶段可降级为 BullMQ 直接投递，不作为本地开发的必需依赖。仅在需要完整事件驱动链路时启动。

### 3.3 启动顺序
推荐顺序：
1. 数据库与缓存
2. 消息队列与向量/对象存储
3. server-gateway
4. Worker
5. 前端 web-client

### 3.4 本地验证入口
- 网关健康检查：`GET /health`
- 任务链路观察：SSE / 任务页面
- 数据迁移状态：Prisma migration status

## 4. 环境变量配置

### 4.1 基础变量
| 变量 | 说明 |
|---|---|
| `NODE_ENV` | 运行环境 |
| `PORT` | 网关端口 |
| `GLOBAL_API_PREFIX` | API 前缀，通常为 `/api` |

### 4.2 数据库与缓存
| 变量 | 说明 |
|---|---|
| `DATABASE_URL` | PostgreSQL 连接串 |
| `REDIS_HOST` | Redis 主机 |
| `REDIS_PORT` | Redis 端口 |
| `KAFKA_BROKERS` | Kafka broker 列表 |

### 4.3 向量与对象存储
| 变量 | 说明 |
|---|---|
| `QDRANT_HOST` | Qdrant 主机 |
| `QDRANT_PORT` | Qdrant 端口 |
| `QDRANT_COLLECTION_ASSETS` | 向量集合名 |
| `MINIO_ENDPOINT` | MinIO 主机 |
| `MINIO_PORT` | MinIO 端口 |
| `MINIO_ACCESS_KEY` | MinIO 访问键 |
| `MINIO_SECRET_KEY` | MinIO 密钥 |
| `MINIO_BUCKET_NAME` | 资源桶名 |

### 4.4 AI 与限流参数
| 变量 | 说明 |
|---|---|
| `VOLC_ARK_API_URL` | 火山方舟 API 地址 |
| `VOLC_ARK_API_KEY` | 火山方舟 Key，占位或密钥注入 |
| `VOLC_ARK_DOUBAO_PRO_ENDPOINT` | 文本模型端点 |
| `VOLC_ARK_DOUBAO_VIDEO_ENDPOINT` | 视频模型端点 |
| `VOLC_ARK_DOUBAO_RATE_LIMIT_RPM` | 文本模型限流 |
| `VOLC_ARK_SEEDANCE_MAX_CONCURRENCY` | 视频模型并发上限 |

### 4.5 安全要求
- 生产密钥不得写入仓库。
- 演示环境使用占位值或密钥管理注入。
- 所有敏感变量仅允许在受控环境中配置。

## 5. 本地启动与关停

### 5.1 推荐启动命令顺序
1. 启动基础依赖服务。
2. 执行数据库迁移。
3. 启动 `apps/server-gateway`。
4. 启动 `workers/gpu-slicing-worker`。
5. 启动 `workers/remotion-render-worker`。
6. 启动 `apps/web-client`。

### 5.2 关停顺序
建议逆序关停：
1. 前端
2. Worker
3. 网关
4. 依赖服务

### 5.3 启动后检查
- `GET /health`
- 任务创建与状态查询
- 迁移状态检查
- 对象存储连通性
- Redis 连通性

## 7. CI/CD 流程

### 7.1 流水线门禁
CI/CD 属于 P2 加分项，`.github/workflows/tks_production_ci_cd.yml` 已配置基础流水线骨架，随项目推进逐步启用以下门禁：
- `pnpm install --frozen-lockfile`（已启用；失败时不允许回退到普通 install）
- `pnpm openapi:lint`（已启用）
- `pnpm lint`（P2 阶段逐步启用）
- `pnpm typecheck`（P2 阶段逐步启用）
- `pnpm test`（P2 阶段逐步启用）
- Docker 镜像构建（P2 阶段逐步启用）
- `/health` 烟测（P2 阶段逐步启用）
- Prisma migration status 检查（P2 阶段逐步启用）

### 7.2 分支触发
- push 到 `main`、`develop`
- pull request 到 `main`

### 7.3 CI 环境
- `ubuntu-latest`
- Node.js 22
- pnpm 9

## 8. 发布后验证

### 8.1 基础验证
1. `GET /health` 返回正常。
2. 数据库迁移状态正常。
3. 前端页面能正常打开。
4. 任务接口能正常创建并观察状态。

### 8.2 联调验证
- 上传素材链路可用
- 剧本生成链路可用
- 一键成片链路可用
- 分析看板查询可用

### 8.3 发布失败处理
- 先停止继续扩散发布。
- 检查镜像是否构建成功。
- 检查 Docker Compose 服务状态。
- 检查数据库迁移。
- 检查依赖服务连通性。

---

## 9. 回滚策略

### 9.1 代码回滚
- 回退到上一个稳定 Git 版本。
- 重新构建镜像并通过 Docker Compose 启动。

### 9.2 镜像回滚
- 切换到前一个已验证镜像 tag。
- 重新执行 Docker Compose 启动。

### 9.3 数据回滚
- 迁移异常时按迁移记录处理。
- 必要时优先恢复数据库与任务状态一致性。

---

## 10. 验证标准

1. 本地是否可完成依赖安装与启动。
2. 环境变量是否齐全且不泄漏敏感值。
3. 镜像是否可构建、可推送、可发布。
4. 部署后验证是否可执行。
5. 回滚路径是否明确且可操作。
